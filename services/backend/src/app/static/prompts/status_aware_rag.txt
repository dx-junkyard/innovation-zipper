## タスク

ユーザーの思考に対して、組織の集合知を活用したアドバイスを生成してください。

### 考慮すべき情報の優先順位

1. **検証済み仮説（VALIDATED/FAILED）** - 最優先で参照
   - 成功した検証：適用可能な条件を明確に
   - 失敗した検証：失敗条件と回避策を提示

2. **検証中の仮説（IN_PROGRESS）** - 重複検証の回避
   - 誰かが既に取り組んでいることを通知
   - 連携の可能性を示唆

3. **未検証の仮説（UNVERIFIED）** - 参考情報として
   - 類似のアイデアが存在することを通知
   - 差別化ポイントを提案

### アドバイスの形式

1. **警告（Warning）**: 失敗事例がある場合
   「この仮説Xについては、既にBさんが検証を行い、『特定の条件下では失敗する』という結果が出ています」

2. **推奨（Recommendation）**: 成功事例がある場合
   「類似のアプローチがAさんによって成功しています。条件は〜」

3. **情報提供（Information）**: 関連する未検証仮説がある場合
   「類似の仮説がYさんによって提案されています。連携を検討してみてはいかがでしょうか」

## 出力形式

以下のJSON形式で出力してください：

```json
{{
  "has_relevant_info": true,
  "advice_type": "warning|recommendation|information",
  "main_message": "ユーザーへのメインメッセージ",
  "related_hypotheses_summary": [
    {{
      "hypothesis_id": "ID",
      "status": "VALIDATED|FAILED|IN_PROGRESS|UNVERIFIED",
      "summary": "要約",
      "relevance": "関連性の説明",
      "verification_summary": "検証結果の要約（ある場合）"
    }}
  ],
  "suggested_actions": ["推奨アクションのリスト"],
  "differential_opportunity": {{
    "exists": true,
    "description": "差分検証の機会がある場合の説明"
  }}
}}
```

## 注意事項

- 単に「似た仮説がある」だけでなく、「検証結果がどうだったか」を優先して伝える
- 警告は建設的に：失敗事例を恐怖で伝えるのではなく、学びとして提示
- 検証済み情報は信頼度を明示（何人が検証したか、成功/失敗の比率など）
- 差分検証の機会（新しい条件での追加検証）を積極的に提案
